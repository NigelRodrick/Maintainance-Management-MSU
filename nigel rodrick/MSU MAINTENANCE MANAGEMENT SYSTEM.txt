2. Target Architecture
2.1 Architecture Style Decision
Recommended: Modular Monolith
Justification: The system serves a single institution (Midlands State University) with a user base that does not warrant the operational overhead of a microservices deployment. A modular monolith retains the simplicity of a single deployable unit whilst enforcing strict internal module boundaries through well-defined interfaces. This enables future extraction into microservices if scale demands it, without the premature complexity that microservices would introduce at this stage. The Flask Blueprint architecture already provides the foundation; the refactor enforces clean contracts between modules.

2.2 Layered Architecture
Layer	Responsibility	Technology
Presentation Layer	HTML templates rendered server-side; Jinja2 templates per role; responsive UI with Bootstrap 5	Jinja2, Bootstrap 5, Plotly.js
API Layer	Versioned REST endpoints (/api/v1/); JSON responses; JWT or session token auth; rate limiting per endpoint	Flask Blueprints, Flask-Limiter
Service Layer	Business logic only; no direct DB calls; all external dependencies injected; stateless functions	Python services, dependency injection
Domain Layer	Core business entities, value objects, status machines, business rules	Python dataclasses / Pydantic models
Data Access Layer	All DB interactions via repository pattern; SQLAlchemy ORM exclusively; no raw PyODBC in services	SQLAlchemy, repository classes

2.3 System Components
2.3.1 Authentication & Authorisation
•	Flask-Login for session management with explicit 30-minute inactivity timeout.
•	Flask-WTF for CSRF protection on all form submissions.
•	Werkzeug password hashing (bcrypt) with minimum 12 rounds.
•	Role enum: admin | supervisor | staff — enforced at decorator and service layer.
•	All role checks via @require_role(Role.ADMIN) decorator pattern, never inline conditionals.

2.3.2 Job Management Engine
•	JobRequest lifecycle state machine: PENDING → IN_PROGRESS → COMPLETED (with CANCELLED branch).
•	State transitions validated in JobService — illegal transitions raise DomainException.
•	Every transition recorded in job_status_history with actor, timestamp, and optional notes.

2.3.3 Assignment Engine
•	WorkerRepository provides availability and skill lookups.
•	AssignmentService enforces one active assignment per worker at a time.
•	Recommendations ranked by: matching category > current workload > historical performance score.

2.3.4 Analytics Engine
•	Separate AnalyticsService reads from dedicated reporting views, never from live transaction tables directly.
•	Phase 1 charts pre-computed and cached in Redis with 5-minute TTL.
•	Phase 3+ ML inference served via a dedicated /api/v1/analytics/predict endpoint.

2.3.5 Notification System
•	Celery task queue (backed by Redis) for async notification dispatch.
•	Initial implementation: in-app notifications stored in notifications table.
•	Phase 7 extension: email notifications via Flask-Mail / SendGrid.

2.3.6 Audit & Logging System
•	UserActivityLog captures all state-changing operations with user, IP, user agent, and timestamp.
•	Structured JSON logging via Python logging to stdout — collected by log aggregator.
•	Separate audit schema tables are read-only to application service account.

2.4 Infrastructure Design
Component	Solution	Configuration
Primary Database	SQL Server 2019 Standard	Connection pooling via SQLAlchemy pool_size=10, max_overflow=20
Caching Layer	Redis 7.x	Flask-Caching + Celery broker. TTL 300s for dashboards, 3600s for reports
Task Queue	Celery + Redis	Workers: report generation, email dispatch, ML inference
Reverse Proxy	Nginx 1.24	TLS termination, static file serving, upstream to Gunicorn (4 workers)
Monitoring	Prometheus + Grafana	Flask metrics via prometheus-flask-exporter. Grafana dashboards for request rate, error rate, DB pool saturation

3. Database & Data Architecture
3.1 Target Schema — Standardised Structure
All tables use lowercase snake_case. Status and role values are UPPERCASE. All FKs are NOT NULL except where business logic genuinely permits no parent (noted below). Soft delete via is_deleted BIT column on all transaction tables.

3.1.1 workers Table (New)
CREATE TABLE workers (
    id              INT           PRIMARY KEY IDENTITY(1,1),
    user_id         INT           NULL,   -- FK to users.id (NULL if worker has no system account)
    full_name       VARCHAR(150)  NOT NULL,
    department      VARCHAR(100)  NOT NULL,
    skill_category  VARCHAR(50)   NOT NULL,  -- electrical|plumbing|carpentry|mechanical|civil
    is_active       BIT           NOT NULL DEFAULT 1,
    created_at      DATETIME2     NOT NULL DEFAULT SYSUTCDATETIME(),
    updated_at      DATETIME2     NOT NULL DEFAULT SYSUTCDATETIME(),
    is_deleted      BIT           NOT NULL DEFAULT 0
);

3.1.2 job_requests Table (Revised)
CREATE TABLE job_requests (
    id              INT           PRIMARY KEY IDENTITY(1,1),
    reference_no    VARCHAR(20)   NOT NULL UNIQUE,  -- e.g. JR-2026-00001
    department      VARCHAR(100)  NOT NULL,
    description     NVARCHAR(MAX) NOT NULL,
    category        VARCHAR(50)   NOT NULL,
    priority        VARCHAR(20)   NOT NULL DEFAULT 'MEDIUM',
    status          VARCHAR(20)   NOT NULL DEFAULT 'PENDING',
    submitted_by    INT           NOT NULL,  -- FK to users.id (NOT NULL — must have owner)
    date_created    DATETIME2     NOT NULL DEFAULT SYSUTCDATETIME(),
    updated_at      DATETIME2     NOT NULL DEFAULT SYSUTCDATETIME(),
    is_deleted      BIT           NOT NULL DEFAULT 0,
    CONSTRAINT CK_job_requests_status   CHECK (status   IN ('PENDING','IN_PROGRESS','COMPLETED','CANCELLED')),
    CONSTRAINT CK_job_requests_priority CHECK (priority IN ('LOW','MEDIUM','HIGH','CRITICAL')),
    CONSTRAINT FK_job_requests_users FOREIGN KEY (submitted_by) REFERENCES users(id)
);

3.1.3 assignments Table (Revised)
CREATE TABLE assignments (
    id              INT           PRIMARY KEY IDENTITY(1,1),
    job_id          INT           NOT NULL,
    worker_id       INT           NOT NULL,  -- FK to workers.id (single source of truth)
    status          VARCHAR(20)   NOT NULL DEFAULT 'ASSIGNED',
    start_time      DATETIME2     NULL,
    end_time        DATETIME2     NULL,
    created_at      DATETIME2     NOT NULL DEFAULT SYSUTCDATETIME(),
    is_deleted      BIT           NOT NULL DEFAULT 0,
    CONSTRAINT CK_assignments_dates   CHECK (end_time IS NULL OR end_time >= start_time),
    CONSTRAINT CK_assignments_status  CHECK (status IN ('ASSIGNED','IN_PROGRESS','COMPLETED','CANCELLED')),
    CONSTRAINT FK_assignments_job_requests FOREIGN KEY (job_id) REFERENCES job_requests(id),
    CONSTRAINT FK_assignments_workers      FOREIGN KEY (worker_id) REFERENCES workers(id)
);

3.1.4 materials Table (Revised)
CREATE TABLE materials (
    id                  INT           PRIMARY KEY IDENTITY(1,1),
    job_id              INT           NOT NULL,  -- FK enforced, NOT NULL
    item_name           VARCHAR(150)  NOT NULL,
    unit                VARCHAR(30)   NOT NULL DEFAULT 'units',
    quantity_required   DECIMAL(10,2) NOT NULL,
    quantity_used       DECIMAL(10,2) NOT NULL DEFAULT 0,
    created_at          DATETIME2     NOT NULL DEFAULT SYSUTCDATETIME(),
    CONSTRAINT CK_materials_quantity CHECK (quantity_used <= quantity_required AND quantity_required > 0),
    CONSTRAINT FK_materials_job_requests FOREIGN KEY (job_id) REFERENCES job_requests(id)
);

3.1.5 job_status_history Table (Replaces JobStatusAudit)
CREATE TABLE job_status_history (
    id              INT           PRIMARY KEY IDENTITY(1,1),
    job_id          INT           NOT NULL,
    from_status     VARCHAR(20)   NULL,
    to_status       VARCHAR(20)   NOT NULL,
    changed_by      INT           NOT NULL,
    changed_at      DATETIME2     NOT NULL DEFAULT SYSUTCDATETIME(),
    notes           NVARCHAR(500) NULL,
    CONSTRAINT FK_jsh_job_requests FOREIGN KEY (job_id) REFERENCES job_requests(id),
    CONSTRAINT FK_jsh_users        FOREIGN KEY (changed_by) REFERENCES users(id)
);

3.3 Migration Strategy (Zero Data Loss)
1.	Create a full backup of CentralServices_AM_DB before any migration step.
2.	Run migration scripts in a transaction — each script begins with BEGIN TRANSACTION and only commits after validation.
3.	Migrate status values: UPDATE job_requests SET status = UPPER(TRIM(status)) WHERE status NOT IN ('PENDING','IN_PROGRESS','COMPLETED','CANCELLED');
4.	Migrate role values: UPDATE users SET role = LOWER(TRIM(role));
5.	Create workers table; populate from DISTINCT (worker_name, worker_id) in assignments; update assignments.worker_id FK; drop worker_name column.
6.	Set submitted_by NOT NULL after backfilling any NULL rows to a designated 'system' user account.
7.	Rename JobStatusAudit → job_status_history with column renames.
8.	Add soft-delete columns to all transaction tables (ALTER TABLE … ADD is_deleted BIT NOT NULL DEFAULT 0).
9.	Apply all new CHECK constraints.
10.	Drop the predictions table after archiving its contents to a CSV export for historical record.
11.	Run integrity validation script: confirm zero orphaned records, zero constraint violations.

9. Analytics & ML Roadmap
9.1 Architecture Integration
The analytics capability is implemented as a distinct AnalyticsModule within the modular monolith. It reads exclusively from reporting views (vw_*) and never writes to transaction tables. ML models are trained offline via Jupyter notebooks and served as serialised artefacts through an MLModelService.

9.2 Phased Analytics Delivery
Phase	Type	Deliverables	Data Sources
A1	Descriptive Analytics	Interactive dashboards: jobs by category, status, department. Mean resolution time. Worker throughput. Daily/weekly/monthly trend lines. Excel/CSV export.	job_requests, assignments, job_status_history. Views: vw_department_summary, vw_worker_performance.
A2	Diagnostic Analytics	Root cause analysis: which departments generate repeated jobs of same category. Correlation heatmaps: priority vs resolution time. Worker skill-to-category mismatch identification.	Historical job_requests + assignments + materials. Pandas EDA in Jupyter. Seaborn correlation matrices.
A3	Predictive ML	Model 1: Priority Classifier (Random Forest — features: category, department, description keywords). Model 2: Resolution Time Estimator (Gradient Boosting — features: priority, category, worker skill, historical avg). Model 3: Technician Matcher (k-NN — features: skill_category, current_workload, past_performance_score).	Training: simulated_data.csv + production data (post Phase A2). Scikit-learn pipeline. Model artefacts stored in /models/ (versioned). Served via MLModelService.
A4	Prescriptive Optimisation	Automated assignment recommendations with confidence score displayed to supervisor before confirmation. Job escalation engine: if resolution_time_estimate > SLA_threshold → auto-escalate priority. Queue optimisation: sort pending jobs by expected completion impact.	Models from A3 + business rule engine. AssignmentService extended with recommend_worker() method. SLA thresholds configurable in admin settings.
A5	Visualisation	Plotly Dash embedded dashboards. Interactive filtering by date range, department, category, worker. Exportable charts (PNG/SVG). Infographic-style summary cards for management reporting.	Plotly.js in templates. Pre-computed chart data served from Redis cache. Dash app mounted at /analytics/dashboard.
A6	Recommendations Engine	Admin-facing recommendation feed: 'Department X has 3 recurring plumbing faults — consider preventive maintenance'. Worker training gap recommendations based on skill vs demand mismatch. Budget forecast: predicted material spend by category for next quarter.	Rule-based engine initially. A3 models feed probabilistic recommendations. Recommendations stored in recommendations table for audit and feedback loop.

9.3 Data Pipeline Design
•	Training data pipeline: Jupyter notebooks (EDA → Feature Engineering → Model Training → Evaluation) → serialised model artefacts to /models/{model_name}_v{version}.pkl.
•	Inference pipeline: JobRequest created → AnalyticsService.classify(request) → MLModelService.predict(features) → confidence score stored with request.
•	Model versioning: each model stored with version number; MLModelService loads the 'current' version from a config table in the database.
•	Retraining trigger: scheduled Celery task runs monthly; if new training data exceeds 500 records since last training, retraining job is enqueued.
•	Model lifecycle: draft → staging (A/B test against rule-based) → production (if accuracy > rule-based) → deprecated.
